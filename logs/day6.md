日志 Day6：2026年6月12日

今日工作概述

  今天在前几天完成核心功能开发和界面设计的基础上，对系统进行了深入的技术细节优化和前端体验打磨。主要集中在侧边栏交互改进、个人中心Tab页面加载体验优化、购物车数量修改的库存校验逻辑、订单时间轴节点的完善、管理后台批量操作的数据一致性处理、以及部分后端接口的性能优化。这些改进涉及面较广，从顾客端到管理端都有覆盖，整体上让系统的体验更加流畅和完整。

一、侧边栏交互优化

  1. 个人中心子菜单展开逻辑

  之前侧边栏中个人中心使用了 Element Plus 的 el-sub-menu 组件来实现子菜单展开。但在测试中发现当用户点击个人中心后跳转到 Profile 页面时，如果页面刷新或者用户通过浏览器地址栏直接访问 Profile 页面，子菜单会默认收起，用户无法直观地看到当前处于个人中心的哪个子模块。修改后在 App.vue 中通过 currentRoute 计算属性动态判断当前路由是否属于个人中心模块，如果是就将子菜单的 default-openeds 设置为展开状态，确保用户在任何情况下都能看到子菜单的展开状态。

  2. 购物车角标动态刷新

  购物车菜单项旁边的数量角标之前只在页面首次加载时获取数据，但如果用户在商品详情页加入购物车后通过浏览器的返回按钮回到商品列表，或者在其他标签页操作了购物车，角标不会自动更新。修改后在路由变化时增加了对购物车数量的重新获取逻辑，当用户导航到购物车相关或商品相关页面时自动刷新角标数据。

  3. 侧边栏用户头像昵称截取

  侧边栏顶部的用户头像区域之前使用用户名的第一个字符作为头像文字。但如果用户名为空或只有一个字符时会显示异常。修改后增加了对空值的兜底处理，用户名为空时显示字母 U 作为默认头像文字。

二、个人中心页面改进

  1. Tab 切换时数据的按需加载

  之前个人中心页面在加载时会一次性请求基本信息、收货地址和收藏列表三个接口的数据。如果用户只是想要修改密码，根本不需要加载地址和收藏数据，这样会造成不必要的网络请求和数据库查询。修改后改为按需加载模式，只有当用户点击对应 Tab 时才去请求该 Tab 需要的数据。初始进入页面时根据 URL 参数中的 Tab 值来决定加载哪些数据。如果 URL 中没有指定 Tab 参数，只加载基本信息。这个改动减少了首屏加载时间，也减轻了后端数据库的查询压力。

  2. 头像上传的图片格式校验

  之前头像上传功能没有对文件类型进行限制，用户可能选择非图片格式的文件导致显示异常。修改后在文件选择回调中增加了对文件类型的校验，只允许常见的图片格式如 JPEG、PNG、GIF 和 WebP。同时增加了对文件大小的限制，超过两兆的图片会提示用户压缩后再上传，避免 Base64 编码后的数据过大导致数据库存储和网络传输的性能问题。

  3. 修改密码后的状态重置

  用户修改密码成功后，之前只是显示成功提示，但表单中已经填写的密码字段没有被清空。如果用户接着再次修改密码，会看到之前填写的旧数据残留在表单中。修改后在密码修改成功后自动清空三个密码输入框，并将表单的验证状态重置为初始状态。

三、购物车功能完善

  1. 数量修改时的库存校验

  购物车中修改商品数量时，之前没有对库存上限进行严格校验。如果用户将数量改为一个超过当前库存的值，提交订单时才会在后端报错，用户体验不好。修改后在数量修改的前端逻辑中增加了实时校验，数量输入框的最大值绑定到该商品当前库存量，用户无法输入超过库存的数量。如果用户试图通过直接输入数字的方式填入超限值，输入框会自动修正为最大可购买数量。

  2. 地址选择后的字段锁定

  用户从已有地址列表中选择了地址后，邮寄信息中的收件人、联系电话和地址三个字段会自动填充。但之前填充后用户仍然可以手动修改这些字段，如果改了之后又切换地址，可能会造成数据混淆。修改后当用户选择了已有地址时，三个字段变为只读状态并显示灰色背景，明确告知用户这些字段来自已保存的地址。如果用户需要修改，可以先清空地址选择，然后手动填写新的邮寄信息。

  3. 运费计算的前端实时预览

  运费计算规则为首重一公斤内包邮、超出一公斤每公斤加收八元。之前在结算区只显示了最终的运费金额，没有展示计算过程。修改后在运费区域增加了计算明细的展示，用户可以看到自己的货物重量、是否超出包邮范围、以及超出部分的具体收费计算，让费用构成更加透明。

四、订单相关功能优化

  1. 订单时间轴节点的完善

  之前的订单时间轴在某些状态下缺少关键节点。例如已取消的订单没有区分是顾客自行取消还是管理员取消。已完成的订单没有标注收货确认的时间。修改后完善了时间轴的节点逻辑，顾客自行取消显示取消人和取消时间，管理员取消显示操作人和取消原因。已收货的订单在时间轴中展示确认收货的具体时间。退款中的订单展示退款申请的时间和审核状态节点。

  2. 订单列表筛选条件的持久化

  用户在订单列表中选择了状态筛选或输入了搜索关键词后，如果点击进入某个订单的详情页再返回列表，之前设置的筛选条件会丢失，用户需要重新筛选。修改后在 URL 参数中保留了筛选条件，用户从详情页返回时列表会自动恢复到之前的筛选状态。

  3. 支付弹窗的费用明细展示

  支付确认弹窗之前只显示了总金额，没有展示金额构成。修改后增加了费用明细的展示，包括商品金额、运费和实付金额三个部分，让用户在付款前能清楚了解费用的具体构成。商品金额和运费分别展示，最后汇总为实付金额，格式清晰一目了然。

五、后端接口优化

  1. 商品列表接口的查询性能优化

  商品列表接口需要同时支持分类筛选、价格区间过滤、排序和分页，SQL 查询的条件组合比较复杂。之前的实现是根据前端传来的参数动态拼接查询条件，但每次请求都会执行一次统计总数的查询和一次分页数据的查询。修改后将统计查询和分页查询中的公共筛选条件提取为独立的查询构建函数，避免了重复的条件拼接逻辑，也让代码更易于维护。

  2. 订单创建接口的事务处理

  创建订单时需要同时写入订单主表、订单明细表，还需要清空购物车缓存，涉及数据库写入和缓存删除两个不同的存储系统。之前如果数据库写入成功但缓存删除失败，会出现订单已创建但购物车没有被清空的情况。修改后将缓存删除操作放在了数据库事务提交之后执行，如果数据库写入失败则不会删除缓存，避免了数据不一致的问题。

  3. 管理后台统计查询的日期范围优化

  仪表盘的销售趋势统计之前查询的是全部历史数据，随着数据量增长查询会越来越慢。修改后限制为只查询最近三十天的数据，通过日期范围过滤减少了扫描的数据量。同时为订单表的订单日期字段添加了索引，加快了按日期范围查询的速度。

六、文件改动清单

  今天的改动主要是对已有功能的细节打磨，不涉及新增文件。前端修改了六个 Vue 组件文件，包括 App.vue 的侧边栏交互逻辑、Profile.vue 的 Tab 按需加载和头像校验、Cart.vue 的库存校验和地址锁定逻辑、Orders.vue 的筛选条件持久化、OrderDetail.vue 的时间轴完善。后端修改了三个路由文件，优化了商品列表查询的代码结构、订单创建的缓存处理和管理后台统计查询的日期范围限制。

七、下一步计划

  系统的核心功能已经比较完整，后续的改进方向主要集中在细节体验的打磨和性能优化上。还有一些可以考虑的功能点，比如为商品列表增加更丰富的筛选维度如按厂家筛选、增加订单的物流追踪展示、完善管理员端的操作日志查询功能等。具体做什么将根据后续的反馈和剩余时间来灵活安排，优先做对用户体验影响最大的改进。
